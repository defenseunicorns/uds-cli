## What You'll Accomplish

Bundle overrides let you customize Helm charts inside Zarf packages without modifying the packages themselves. By the end of this guide you'll know how to:

- Override static Helm values at bundle-creation time
- Define runtime-configurable variables
- Use `uds-config.yaml` to manage configuration
- Understand variable precedence across all override sources

---

## Prerequisites

- [UDS CLI installed](/cli/getting-started/installation/)
- A `uds-bundle.yaml` referencing at least one Zarf package with Helm charts

---

## Overview

The `overrides` block in `uds-bundle.yaml` maps to the structure of a Zarf package:

```yaml
overrides:
  <component-name>:    # Zarf component containing the chart
    <chart-name>:      # Helm chart name within that component
      namespace: ...
      valuesFiles: [...]
      values: [...]
      variables: [...]
```

## Steps

<Steps>

1. **Add static value overrides**

   Use the `values` key to set Helm values that are fixed at bundle-creation time and **cannot be changed** at deploy time:

   ```yaml title="uds-bundle.yaml"
   packages:
     - name: helm-overrides-package
       path: "path/to/pkg"
       ref: 0.0.1
       overrides:
         helm-overrides-component:
           podinfo:
             values:
               - path: "replicaCount"
                 value: 2
               - path: "podinfo.tolerations"
                 value:
                   - key: "unicorn"
                     operator: "Equal"
                     value: "defense"
                     effect: "NoSchedule"
   ```

   The `path` uses dot notation to locate the value in the chart's `values.yaml`.

2. **Add a values file**

   For bulk overrides, provide a YAML file via `valuesFiles`:

   ```yaml title="uds-bundle.yaml"
   overrides:
     helm-overrides-component:
       podinfo:
         valuesFiles:
           - overrides/podinfo-values.yaml
         values:
           - path: "replicaCount"
             value: 2
   ```

   ```yaml title="overrides/podinfo-values.yaml"
   podAnnotations:
     customAnnotation: "customValue"
   ui:
     message: "Hello from bundle"
   ```

   When multiple `valuesFiles` are listed, later files take precedence over earlier ones. Static `values` take precedence over all `valuesFiles`.

3. **Add runtime-configurable variables**

   Unlike `values`, `variables` can be overridden at deploy time:

   ```yaml title="uds-bundle.yaml"
   overrides:
     helm-overrides-component:
       podinfo:
         variables:
           - name: UI_COLOR
             path: "ui.color"
             description: "Set the color for podinfo's UI"
             default: "purple"
           - name: SECRET_VAL
             path: "testSecret"
             description: "A sensitive value, masked in output"
             sensitive: true
           - name: TLS_CERT
             path: "tls.cert"
             description: "Path to a certificate file"
             type: file
   ```

   Variable types:
   - `raw` (default) ŌĆö string, int, map, etc.
   - `file` ŌĆö the variable resolves to the **contents** of the referenced file

   > [!CAUTION]
   > If a variable accepts a file value but is missing `type: file`, the file will not be processed.

4. **Override a variable at deploy time**

   Three methods, in order of increasing precedence:

   **Via `uds-config.yaml`** (place in the same directory as the bundle):

   ```yaml title="uds-config.yaml"
   variables:
     helm-overrides-package:
       ui_color: green   # variable names are case-insensitive
   ```

   **Via environment variable** (prefix with `UDS_`):

   ```bash
   UDS_UI_COLOR=green uds deploy uds-bundle-helm-overrides.tar.zst --confirm
   ```

   **Via `--set` flag**:

   ```bash
   # Apply to all packages
   uds deploy uds-bundle-helm-overrides.tar.zst --set ui_color=green

   # Apply to a specific package only
   uds deploy uds-bundle-helm-overrides.tar.zst --set helm-overrides-package.ui_color=green
   ```

   > [!CAUTION]
   > Helm override variables and Zarf variables share the same `--set` syntax. Avoid naming conflicts between them.

5. **Configure global options with `uds-config.yaml`**

   The `uds-config.yaml` supports global CLI options, shared variables, and per-package variables:

   ```yaml title="uds-config.yaml"
   options:
     log_level: debug
     architecture: arm64
     no_log_file: false
     no_progress: false
     uds_cache: /tmp/uds-cache
     tmp_dir: /tmp/tmp_dir
     insecure: false
     oci_concurrency: 3

   shared:
     domain: uds.dev  # applied to all packages in the bundle

   variables:
     my-zarf-package:
       ui_color: green
       ui_msg: "Hello Unicorn"
   ```

   Place the file in your working directory or set its path via the `UDS_CONFIG` environment variable.

6. **Share variables across packages**

   Use `exports` and `imports` to pass a variable from one package to another:

   ```yaml title="uds-bundle.yaml"
   packages:
     - name: output-var
       repository: localhost:888/output-var
       ref: 0.0.1
       exports:
         - name: OUTPUT

     - name: receive-var
       repository: localhost:888/receive-var
       ref: 0.0.1
       imports:
         - name: OUTPUT
           package: output-var
   ```

   For variables shared across all packages without import/export, use the `shared` key in `uds-config.yaml` or a `UDS_`-prefixed environment variable.

7. **Override a Helm chart's namespace**

   ```yaml title="uds-bundle.yaml"
   overrides:
     podinfo-component:
       unicorn-podinfo:
         namespace: custom-podinfo
         values:
           - path: "replicaCount"
             value: 1
   ```

</Steps>

## Variable Precedence

From least to most specific:

| Source | Precedence |
|---|---|
| Declared in Zarf package | Lowest |
| `import`'ed from another package's `export` | |
| `shared` key in `uds-config.yaml` | |
| `variables` key in `uds-config.yaml` | |
| `UDS_` prefixed environment variable | |
| `--set` flag | Highest |

For override `values` specifically:

| Source | Precedence |
|---|---|
| Earlier `valuesFile` | Lowest |
| Later `valuesFile` | |
| Static `values` in `overrides` block | Highest |

## Verify

View all configurable overrides and variables for a bundle before deploying:

```bash
uds inspect --list-variables uds-bundle-<name>.tar.zst
```

Run `uds deploy` without `--confirm` to see the pre-deploy summary showing all currently set values.

## Troubleshooting

| Symptom | Check |
|---|---|
| Override has no effect | Verify the `<component>` and `<chart>` names match those in the Zarf package's `zarf.yaml` |
| File variable not applied | Ensure the variable has `type: file` set |
| Sensitive value visible in output | Confirm `sensitive: true` is set on the variable |
| File path not resolving | Relative file paths are resolved relative to the `uds-config.yaml` directory |

## Duplicate Packages

You can deploy multiple instances of the same Zarf package in a bundle by giving each a unique `name` and deploying them into different namespaces using `overrides`. This requires the `namespace` override on any Helm charts in those packages.

For full details and examples, see [duplicate packages in the repository](https://github.com/defenseunicorns/uds-cli/blob/main/dev-docs/reference/CLI/quickstart-and-usage.md#duplicate-packages-and-naming).
